---

Preciso que você construa um **Módulo de Compras completo** no meu ERP (React + Express + tRPC + Drizzle ORM + MySQL). Leia todo este documento antes de começar. Não pule nenhum item.

---

## CONTEXTO DO SISTEMA

- Stack: React 19, TypeScript, tRPC v11, Drizzle ORM, MySQL, shadcn/ui, Tailwind
- Já existem módulos: RH, SST, Jurídico, Obras, Orçamento
- O módulo de Orçamento já tem a **planilha meta** com campos: `venda`, `custo` e `meta` por item da EAP, vinculada a cada obra
- Cada obra tem uma EAP (Estrutura Analítica do Projeto) com insumos orçados
- Usuários têm login individual e papel (role): `admin_master`, `admin`, `user`
- Cada usuário (engenheiro/aux_adm) é vinculado a uma ou mais obras

## O QUE CONSTRUIR

### 1. MÓDULO ALMOXARIFADO CENTRAL
Criar antes de compras, pois compras depende dele.

**Tabelas:**
```sql

  id, nome, unidade, quantidade_atual, quantidade_minima,
  categoria, codigo_interno, observacoes,
  created_at, updated_at

almoxarifado_movements (
  id, item_id, tipo ENUM('entrada','saida','reserva','cancelamento'),
  quantidade, obra_id, solicitacao_id,
  motivo, usuario_id, usuario_nome,
  created_at
)
```

**Funcionalidades:**
- CRUD de itens do estoque
- Entrada e saída de materiais com histórico
- Alerta visual quando quantidade abaixo do mínimo
- Ao criar uma Solicitação de Compra: **o sistema consulta o almoxarifado automaticamente**
  - Se tiver estoque suficiente → bloqueia compra e obriga retirada do almoxarifado
  - Se tiver estoque parcial → permite pedir apenas a diferença
  - Se não tiver → libera a solicitação normalmente

### 3. MÓDULO DE COMPRAS — FLUXO PRINCIPAL

#### 3.1 Solicitação de Compra (SC)

**Tabela:**
```sql
purchase_requests (
  id, obra_id, eap_item_id, solicitante_id, solicitante_nome,
  status ENUM('rascunho','aguardando_aprovacao','aprovada','recusada','em_cotacao','concluida','cancelada'),
  emergencial BOOLEAN DEFAULT false,
  justificativa_emergencial TEXT,
  prazo_necessidade DATE,
  justificativa_recusa TEXT,
  aprovador_id, aprovador_nome, aprovado_em,
  created_at, updated_at
)

purchase_request_items (
  id, solicitacao_id, insumo_nome, unidade, quantidade,
  quantidade_estoque_disponivel,
  quantidade_retirada_estoque,
  quantidade_a_comprar,
  valor_meta_unitario DECIMAL,  -- vem da planilha meta do orçamento
  observacoes
)
```

**Regras:**
- O solicitante **só vê as obras vinculadas ao seu usuário**
- Ao selecionar o insumo: sistema mostra o valor meta da planilha do orçamento
- Ao selecionar o insumo: sistema cruza com almoxarifado e informa disponibilidade
- Toggle **"⚡ COMPRA EMERGENCIAL"** bem visível — se marcado, campo de justificativa obrigatório
- SC emergencial vai para o **topo da fila de aprovação e do comprador**

#### 3.2 Aprovação

**Regras:**
- Aprovador é o engenheiro responsável da obra (configurável)
- Compara o valor solicitado com o **valor meta da planilha do orçamento**:
  - Dentro da meta → pode aprovar normalmente
  - Acima da meta (mesmo R$ 0,01) → alerta visual de estouro, aprovação com confirmação explícita, escala para superior
- Aprovador pode: Aprovar ✅ ou Recusar ❌ com justificativa obrigatória
- SC emergencial: badge vermelho ⚡, notificação imediata, prazo de resposta menor

#### 3.3 Mapa de Cotação

**Tabelas:**
```sql
quotations (
  id, solicitacao_id, status ENUM('aberta','encerrada','cancelada'),
  minimo_fornecedores INT DEFAULT 3,
  fornecedor_vencedor_id,
  comprador_id, comprador_nome,
  created_at, updated_at
)

quotation_suppliers (
  id, cotacao_id, supplier_id, supplier_nome,
  status ENUM('aguardando','respondida','recusada'),
  valor_unitario DECIMAL, valor_total DECIMAL,
  prazo_entrega INT,  -- dias
  condicao_pagamento VARCHAR,
  observacoes,
  respondido_em
)
```

**Funcionalidades:**
- Compras abre cotação com mínimo 3 fornecedores do cadastro
- Sistema exibe **histórico de preços** daquele insumo nos fornecedores
- Após preencher: **ranking automático** por menor preço total
- Comprador define o vencedor (pode não ser o mais barato — pode ter justificativa)

#### 3.4 Ordem de Compra (OC)

**Tabelas:**
```sql
purchase_orders (
  id, solicitacao_id, cotacao_id, supplier_id, supplier_nome,
  obra_id, comprador_id, comprador_nome,
  status ENUM('emitida','aprovada','entregue_parcial','entregue_total','cancelada'),
  valor_total DECIMAL,
  forma_pagamento VARCHAR,
  prazo_entrega DATE,
  condicao_entrega TEXT,
  observacoes,
  emitida_em, created_at, updated_at

purchase_order_items (
  id, ordem_id, insumo_nome, unidade,
  quantidade_pedida, quantidade_recebida DEFAULT 0,
  valor_unitario, valor_total,
  valor_meta_unitario  -- referência da planilha meta
)
```

**Funcionalidades:**
- Gerada a partir do vencedor da cotação
- Ao emitir OC → **gera automaticamente um compromisso financeiro** (contas a pagar)
- Alerta automático se `prazo_entrega` vencer sem recebimento total
- PDF da OC para envio ao fornecedor

#### 3.5 Recebimento na Obra

**Tabelas:**
```sql
receipts (
  id, ordem_id, obra_id, recebedor_id, recebedor_nome,
  status ENUM('parcial','total'),
  nota_fiscal_numero VARCHAR,
  nota_fiscal_url TEXT,  -- upload do arquivo
  observacoes,
  recebido_em, created_at
)

receipt_items (
  id, recebimento_id, ordem_item_id,
  insumo_nome, unidade,
  quantidade_pedida, quantidade_recebida,
  quantidade_pendente
)
```

**Regras:**
- Recebimento parcial:
  - OC atualizada automaticamente com quantidades recebidas
  - Alerta para: solicitante, aprovador, comprador, gestor
  - Pagamento liberado **apenas proporcional ao que chegou**
  - OC permanece aberta aguardando saldo
- Recebimento total: OC encerrada, pagamento liberado integralmente
- Obrigatório anexar nota fiscal

### 6. COMISSÃO DO COMPRADOR

**Tabela:**
```sql
buyer_commissions (
  id, obra_id, obra_nome,
  comprador_id, comprador_nome,
  valor_meta_total DECIMAL,
  valor_comprado_total DECIMAL,
  economia_total DECIMAL,
  percentual_participacao DECIMAL,  -- % das OCs que esse comprador atendeu
  valor_comissao DECIMAL,  -- economia × participação × 20%
  status ENUM('em_aberto','calculada','aprovada_diretor','paga','cancelada'),
  aprovado_por VARCHAR, aprovado_em DATETIME,
  observacoes TEXT,
  calculado_em DATETIME, created_at
)
```

**Regras:**
- Calculada **somente no encerramento da obra**
- Fórmula: `Comissão = (Meta Total - Gasto Total) × 20%`
- Se obra estoura a meta → comissão zero (empresa teve prejuízo)
- Mais de 1 comprador → divide proporcional ao **valor das OCs que cada um atendeu dentro da meta**
  ```
  Comprador A: atendeu R$ 90k em OCs → 90/(90+10) = 90% da comissão
  Comprador B: atendeu R$ 10k em OCs → 10/(90+10) = 10% da comissão
  ```
- Apenas o **Diretor** pode aprovar e liberar o pagamento
- Quando aprovada → gera lançamento no financeiro (contas a pagar para o comprador)

# ⚠️ CORREÇÕES CRÍTICAS — APLICAR ANTES DE QUALQUER COISA

## CORREÇÃO 1 — CATÁLOGO CENTRALIZADO DE INSUMOS

**Problema:** Sem catálogo padrão, o engenheiro digita "cimento" e o comprador digita "Cimento CP-II 50kg" — são registros diferentes. Histórico de preços não funciona, almoxarifado não cruza, relatórios ficam inconsistentes.

**Solução:** Criar catálogo de insumos antes de tudo. Todo item da SC, cotação e OC deve ser selecionado do catálogo — nunca digitado livremente.

```sql
purchase_catalog_items (
  id, companyId,
  nome VARCHAR(255),           -- "Cimento CP-II 50kg"
  nomeAbreviado VARCHAR(100),  -- "Cimento CP-II"
  codigo VARCHAR(50),          -- código interno
  unidade VARCHAR(20),         -- "sc", "kg", "m³", "m²", "un", "l"
  categoria VARCHAR(100),      -- "Argamassa", "Estrutural", "Elétrico"
  ncm VARCHAR(10),             -- código fiscal (para OC)
  descricao TEXT,
  fotoUrl TEXT,
  ativo BOOLEAN DEFAULT true,
  createdAt, updatedAt
)
```

**Tabela de histórico de preços separada (não JSON):**
```sql
supplier_price_history (
  id, companyId,
  catalogItemId INT,           -- item do catálogo
  supplierId INT,              -- fornecedor
  supplierNome VARCHAR(255),
  valorUnitario DECIMAL(10,2),
  unidade VARCHAR(20),
  dataReferencia DATE,         -- data da cotação
  cotacaoId INT,               -- origem do preço
  ordemCompraId INT,
  createdAt TIMESTAMP
)
```

**Regras:**
- ADM cadastra e mantém o catálogo
- Na SC: engenheiro busca do catálogo (autocomplete por nome)
- Na cotação: comprador vê automaticamente o último preço de cada fornecedor para aquele item
- Ao fechar OC: preço é salvo em `supplier_price_history` automaticamente
- Campo de busca com foto do item para facilitar seleção

## CORREÇÃO 3 — CONFIGURAÇÃO DE APROVAÇÃO HIERÁRQUICA

**Problema:** "Acima da meta escala para o superior" foi definido mas não existe onde configurar quem é o superior, quais são os limites por valor, quem aprova o quê.

**Solução:** Tabela de configuração de alçadas de aprovação.

```sql
purchase_approval_rules (
  id, companyId,
  nome VARCHAR(255),           -- "Regra padrão obras"
  obraId INT,                  -- null = vale para todas as obras
  
  -- Nível 1: aprovação normal (dentro da meta)
  nivel1AprovadorTipo ENUM('engenheiro_obra','cargo','usuario_especifico'),
  nivel1AprovadorId INT,       -- userId se tipo = usuario_especifico
  nivel1Cargo VARCHAR(100),    -- se tipo = cargo
  nivel1PrazoHoras INT DEFAULT 24,
  
  -- Nível 2: acima da meta ou valor alto
  nivel2Ativo BOOLEAN DEFAULT true,
  nivel2AprovadorTipo ENUM('gerente','diretor','usuario_especifico'),
  nivel2AprovadorId INT,
  nivel2PrazoHoras INT DEFAULT 8,
  
  -- Compra direta (sem cotação abaixo deste valor)
  limiteCompraDireta DECIMAL(10,2) DEFAULT 500,
  
  -- SLA emergencial (horas para resolver)
  slaEmergencialHoras INT DEFAULT 4,
  
  ativo BOOLEAN DEFAULT true,
  createdAt, updatedAt
)
```

**Como funciona:**
- ADM configura as regras por obra ou geral
- SC criada → sistema consulta a regra da obra → define quem aprova
- Dentro da meta + valor abaixo do limite → aprovação nível 1
- Acima da meta OU valor alto → escala para nível 2
- Abaixo do `limiteCompraDireta` → compra direta sem cotação obrigatória
- SLA emergencial configurável (não hardcoded)

**Tela de configuração (apenas ADM):**

## CORREÇÃO 4 — CANCELAMENTO COM RASTREABILIDADE E ROLLBACK

**Problema:** Não definimos o que acontece quando uma SC ou OC é cancelada. O saldo da meta fica comprometido, o estoque reservado não volta, o financeiro não cancela.

**Solução:** Todo cancelamento deve desfazer todos os efeitos colaterais automaticamente.

**Tabela de log de cancelamentos:**
```sql
purchase_cancellations (
  id, companyId,
  tipo ENUM('sc','oc','cotacao','recebimento'),
  referenciaId INT,            -- id da SC, OC, etc.
  motivo TEXT NOT NULL,
  efeitos JSON,                -- quais rollbacks foram executados
  canceladoPorId INT,
  canceladoPorNome VARCHAR(255),
  canceladoEm TIMESTAMP
)
```

**Rollback automático por tipo:**

```typescript
// Ao cancelar SC:
async function cancelarSC(scId: number, motivo: string, userId: number) {
  const sc = await getSC(scId);
  
  // 1. Liberar reserva no almoxarifado (se havia reserva)
  if (sc.quantidadeRetiradaEstoque > 0) {
    await liberarReservaAlmoxarifado(sc.itemId, sc.quantidadeRetiradaEstoque);
  }
  
  // 2. Atualizar status
  await updateSC(scId, { status: 'cancelada' });
  
  // 3. Notificar solicitante, aprovador e comprador
  await notificarCancelamento(sc, motivo);
  
  // 4. Registrar log
  await registrarCancelamento({ tipo: 'sc', referenciaId: scId, motivo, efeitos: ['liberou_estoque'] });
}

// Ao cancelar OC:
async function cancelarOC(ocId: number, motivo: string, userId: number) {
  const oc = await getOC(ocId);
  
  // 1. Cancelar compromisso financeiro (contas a pagar)
  await cancelarContasPagar(oc.id);
  
  // 2. Liberar saldo comprometido na meta da EAP
  await liberarSaldoMeta(oc.obraId, oc.eapItemId, oc.valorTotal);
  
  // 3. Voltar SC para status 'aprovada' (pode abrir nova cotação)
  await updateSC(oc.solicitacaoId, { status: 'aprovada' });
  
  // 4. Notificar todos os envolvidos
  await notificarCancelamento(oc, motivo);
  
  // 5. Registrar log
  await registrarCancelamento({ tipo: 'oc', referenciaId: ocId, motivo,
    efeitos: ['cancelou_financeiro', 'liberou_saldo_meta', 'reabriu_sc'] });
}
```

**Regras de cancelamento:**
- SC `aguardando_aprovacao` → qualquer um pode cancelar (o próprio solicitante ou ADM)
- SC `aprovada` → só ADM cancela, com motivo obrigatório
- SC `em_cotacao` → só ADM cancela + cancela a cotação vinculada
- OC `emitida` → só ADM + Diretor, com motivo + cancela financeiro
- OC `entregue_parcial` → não pode cancelar (já recebeu material)
- Recebimento → não pode cancelar (apenas ajustar)

## CORREÇÃO 5 — FRETE NA COTAÇÃO E OC

**Problema:** O comprador cotou cimento por R$42/sc mas o frete é R$800 extra. A OC fica com valor errado e o financeiro lança errado.

**Adicionar campos na cotação e OC:**

```sql
-- Adicionar em quotation_suppliers:
  valor_frete DECIMAL(10,2) DEFAULT 0,
  frete_tipo ENUM('cif','fob') DEFAULT 'cif',
  -- CIF = fornecedor paga frete, FOB = comprador paga frete
  valor_total_com_frete DECIMAL(10,2),  -- calculado automaticamente

-- Adicionar em purchase_orders:
  valor_frete DECIMAL(10,2) DEFAULT 0,
  frete_tipo ENUM('cif','fob') DEFAULT 'cif',
  valor_itens DECIMAL(10,2),
  valor_total DECIMAL(10,2),   -- valor_itens + valor_frete
```

**No ranking automático de cotações:** ordenar por `valor_total_com_frete` (não só pelo item).
